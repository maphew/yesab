YESAB Registry Probe · 2026-04-14

How the API could plug into the maps we already built

The current map in out/ is a static Yukon Albers viewer built from the YESAB shapefile zip. The public API is a separate live registry feed with status-rich records, WGS84 point locations, and no advertised pagination. The practical question is whether we use the API to enrich the existing map, build a current-project overlay, or pivot to an API-first product.

3,785 features in the current static map output
385 API projects returned for 2024-2025
176 API projects for 2025 alone
79 API projects for 2026 alone

What the current map already gives us

  • Five local layers from yesab_all.zip: points, lines, placer, polygons, quartz.
  • Joinable identifiers are already present in most layers as ProjectID, Prj_ID, YESAB_PROJ, or Number.
  • Coordinates are projected Yukon Albers values rendered directly in canvas, with no basemap or web dependency.
  • The static site is portable and works over file://, which is valuable for offline distribution.

What the API adds

  • Live registry fields: stage, stageHistory, outcomes, proponent, sectors, districts, governments, decision bodies.
  • Direct record lookup works by either project number or UUID path.
  • Every sampled 2024-2025 record had a locations array and stageHistory.
  • The API carries current workflow state, including daysRemaining on some active stages.

Unadvertised behavior worth using carefully

  • startYear without endYear works as an open-ended query. In practice, startYear=2025 returned 2025 and 2026 projects.
  • Extra query parameters like projectNumber, title, projectId, and stageId were accepted but ignored.
  • Missing project paths return HTTP 200 with a zero-byte body instead of an error.
  • The list endpoint enforces a maximum five-year span difference. 2020-2025 worked, 2020-2026 failed with HTTP 400.

Implication for joining the two sources

The cleanest common key is the project number, not the UUID. The shapefile-derived map already stores project-number style IDs, while the API exposes both project number and UUID. That means we can enrich legacy geometries with live status metadata without first solving a geometry problem.

Static map key: ProjectID / Prj_ID / YESAB_PROJ API key: projectNumber API geometry: locations[].latitude/longitude

Main constraint

The API appears to be point-based. The existing shapefile archive contains richer geometry: lines, polygons, placer extents, quartz extents, and repeated features for one project. So the API is best seen as a live metadata source first, and a geometry source only for recent projects that are missing from the shapefile bundle.

Current map: richer geometry API: fresher status Best combination: join + overlay Sampled 2023 overlap: partial, not complete

Options

These are the four paths that look technically defensible from the probe results.

Option 1

Enrich the existing static map

Lowest risk

Keep the current shapefile geometry exactly as-is and join in API data by projectNumber. Add live stage, outcomes, proponent, stage history, and decision-body detail in the sidebar.

  • Best if we want immediate value without changing the map model.
  • Strong fit because current layers already carry project-number style IDs.
  • Good offline story if we snapshot API responses during build.

Missing piece: recency is limited by how fresh the shapefile zip is.

Option 2

Hybrid map: static geometry plus live current-project overlay

Best balance

Preserve the existing shapefile layers for historical geometry, then add a separate API-fed point overlay for active projects from recent years. That gives us both geometry depth and live operational status.

  • Recent projects missing from the shapefile still show up via API points.
  • Lets us style current stages differently from archived shapes.
  • Avoids pretending API points can replace polygons and linework.

Extra work: convert API WGS84 points into the map’s projected coordinate system.

Option 3

API-first current activity map

Fastest new product

Build a separate map focused only on current and recent registry activity. Use API point locations, stage filters, district views, and a timeline or board for stage progression.

  • Strong if the decision is really about “what is happening now?”
  • Easy to explain and maintain because there is one source of truth.
  • Makes stage-based UX much cleaner than bolting it onto every old geometry layer.

Tradeoff: loses detailed area and corridor geometry from the shapefile archive.

Option 4

Comparison and QA dashboard first

Best if data trust is the issue

Before changing the public-facing map, build a diagnostic view that compares shapefile coverage against API coverage: join success, missing projects, geometry gaps, stale shapes, and stage freshness.

  • Useful if we suspect the zip and registry are drifting apart.
  • Would surface projects that exist in API but not in the shapefile bundle.
  • Creates a normalized data layer we can reuse later for either Option 1, 2, or 3.

Tradeoff: less visible progress at first, but cleaner downstream decisions.

Recommended direction

If the goal is to extend the map we already have rather than replace it, Option 2 is the best next step. It respects the shapefile archive as the geometry backbone and uses the API where it is strongest: live registry status, current projects, and recent additions.

  • Phase 1: normalize and cache API records by projectNumber.
  • Phase 2: join API metadata into existing features and upgrade the details panel.
  • Phase 3: add a recent-project point overlay for records with no local geometry match.

Decision prompts

  • Do we want the map to answer “where is it?” or “what is active now?” first?
  • Should offline portability remain a hard requirement?
  • Is recent-project completeness more important than polygon and corridor geometry?
  • Do we need a data QA pass before we trust automated joins?
Probe notes: live checks were run against https://yesabregistry.ca/api/integration/projects on 2026-04-14. Observed API stage distribution for 2025 projects was led by Decision Document Issued (66), then Recommendation Sent (36), with active workflows such as Adequacy Information Request (15) and Seeking Views and Information (15). Query params beyond startYear and endYear should not be trusted unless we verify them server-side again. A quick 2023 join spot-check against the generated map assets found mixed coverage rather than a full match, which strengthens the case for either Option 2 or Option 4 before relying on automated joins for recent projects.
